I am a senior software engineer who just built xBrain — a Python CLI pipeline that generates, scores, stress-tests, and evolves business ideas using LLMs (Claude API), with meta-learning across runs, adversarial debate, evolutionary operators (mutation, crossover, selection pressure), and persistent memory. I have strong skills in Python, async programming, CLI design, LLM prompt engineering, structured JSON output from LLMs, git automation, and pipeline architecture.

I want to build my next project: an open-ended evolution engine for software itself. Inspired by Gödel machine theory (a self-improving system that can rewrite its own code when it can prove the rewrite is an improvement) and open-ended evolution (systems that continuously produce novel, increasingly complex artifacts without converging to a fixed point).

The core loop:
1. ANALYZE — Read a codebase, understand its structure, metrics, tests, and current weaknesses (code quality, performance, test coverage, missing features, architectural debt)
2. PROPOSE — Use LLMs to generate candidate improvements (bug fixes, refactors, new features, performance optimizations, documentation)
3. IMPLEMENT — Create a git feature branch, apply code changes, commit
4. VERIFY — Run the test suite, linters, benchmarks, type checkers. Measure before/after metrics. Prove the improvement is real and nothing regressed.
5. DECIDE — If verified improvement: merge to main, increment version, document the change. If regression: discard branch. If uncertain: flag for human review.
6. EVOLVE — Feed results back into the system. Learn which types of changes succeed. Adjust strategy. Run the loop again on the new version.

The system should be able to:
- Operate on ANY codebase (not just itself), given a git repo path
- Control git operations: branch creation, commits, merges, version bumps
- Run arbitrary test/build commands configured per-project
- Prove improvement quantitatively (test count, coverage %, lint score, benchmark times, binary size, etc.)
- Document every iteration with before/after metrics
- Eventually improve itself (its own analysis, proposal, and verification capabilities) — the Gödel machine aspect
- Maintain a changelog and release history of autonomous improvements
- Have safety guardrails: human approval gates, rollback capability, scope limits
- The git/build/release pipeline itself is ALSO a target for self-improvement — the engine should be able to improve how it manages branches, commits, and releases

Key insight: this is exactly what we did manually building xBrain — analyzed code, proposed changes, implemented on branches, verified with compile+test+dry-run, committed, bumped version, documented in changelog. The vision is to automate that entire workflow into an autonomous loop that can run on any repo.

Constraints: Python, open source, must work as a CLI tool, must use git as the version control backbone, must produce human-readable documentation of every change it makes. Should start simple (analyze + propose + verify on external repos) before attempting self-modification. Solo developer, evenings and weekends.

What architectures, product concepts, iteration strategies, or novel approaches could make this work? What is the simplest viable starting point that demonstrates the core loop? How should self-improvement be bootstrapped safely?
